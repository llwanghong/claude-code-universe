# 第 15 章：MCP — 通用工具协议

## 为什么 MCP 重要

在 MCP（Model Context Protocol）之前，每个 agent 框架有自己定义和发现工具的方式。开发者必须为每个平台重写工具集成。

MCP 改变了这一点。它是工具发现和调用的标准化协议。为一个 MCP 客户端构建的工具适用于任何 MCP 客户端。Claude Code 实现了一个最完整的 MCP 客户端，支持 8 种传输类型和完整的 OAuth 认证流程。

---

## 八种传输类型

| 传输 | 协议 | 用例 |
|----------|---------|---------|
| stdio | 子进程 stdin/stdout | 本地服务器 |
| SSE | Server-Sent Events | 远程服务器、基于 Web |
| HTTP | REST 风格 | 通用远程 |
| WebSocket | 双向 | 实时服务器 |
| SDK | 进程内 | 嵌入式服务器 |
| IDE v1/v2 | IDE 桥接 | IDE 集成 |
| Claude.ai 代理 | 代理 | 远程连接 |

每种传输类型本质上做相同的事情：产生命名工具的列表，每个工具带有类似于内置 Claude Code 工具的输入 schema。传输类型抽象化了通信协议。

> 💡 **译注**：为什么需要 8 种传输？不同部署场景需要不同通信方式。本地开发用 stdio（最简），远程 Web 服务用 SSE/HTTP，IDE 集成用专用桥接，企业部署用代理。Claude Code 作为"一个产品适配所有场景"的策略，必须在传输层吃下这些复杂性。自己做 Agent 产品，大概率只需要 stdio 和 HTTP。

---

## 工具包装

MCP 集成不只是暴露原始 MCP 工具。它包装它们：Schema 翻译（MCP JSON Schema → Zod）、进度报告、错误处理和结果预算。包装器使 MCP 工具从模型的角度来看与内置工具不可区分。

---

## 动态工具注册

MCP 服务器可以在运行时出现和消失。服务器异步连接——标记为 `pending` 直到完全就绪。工具注册表是动态的：在每次查询轮次中反映当前的工具集，而不是启动时的快照。

---

## OAuth for MCP

一些 MCP 服务器需要认证。Claude Code 实现完整的 MCP OAuth 流程：客户端注册、授权码流程、token 刷新。OAuth 流程是异步的——服务器以 pending 状态开始，认证在后台运行，完成时转变为活跃。

---

## 工具缓存和 Prompt 稳定性

MCP 工具名称按字母顺序排序，追加在内置工具之后。这样添加或移除 MCP 服务器不会移动内置工具的位置，保持缓存前缀逐字节稳定（第 4 章和第 9 章中讨论的同一原则）。

---

## Apply This

**MCP 是通用协议。** 如果你在定义自定义工具协议，你可能在犯错误。为一个 MCP 客户端编写一次工具，它们适用于所有客户端。

**包装外部工具。** 当集成外部工具时，包装它们——不要暴露原始接口。添加进度报告、错误翻译和结果预算。

**异步连接。** 不要在启动时阻塞等待 MCP 服务器。标记为 pending 并在准备好时增量添加工具。

**OAuth 是 MCP 的超能力。** 标准化的 OAuth 支持意味着 MCP 工具可以安全连接到第三方服务而不共享 API 密钥。

**工具对模型来说都是相同的。** 内置、skill 和 MCP 工具对模型来说应该不可区分。统一接口使交换 provider 变得容易。
