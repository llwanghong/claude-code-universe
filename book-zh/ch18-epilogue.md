# 第 18 章：我们学到了什么

## 五个架构赌注

Claude Code 不是唯一的 agentic 系统。它不是第一个。但它做出了区别于 agent 框架版图的五个架构赌注，经过近两千文件和十七个章节，这些赌注值得审视。

### 赌注 1：Generator Loop 超越 Callbacks

大多数 agent 框架给一个流水线：定义工具、注册处理器、让框架编排。开发者写回调。框架决定何时调用它们。

Claude Code 做相反的。`query()` 函数是一个 async generator——开发者拥有循环。模型流式输出响应，generator 产出工具调用，调用者执行它们、追加结果，generator 循环。有一个函数、一个数据流、一个每次交互经过的地方。Generator 返回类型的 10 个终端状态和 7 个继续状态编码了每种可能的结果。循环就是系统。

赌注是单一 generator 函数，即使增长到 1,700 行，将比分布式的回调图更容易理解。在研究了源码后，赌注得到了回报。当你想理解为什么会话结束时，你看一个函数。当你想添加新的终端状态时，你向一个可辨识联合添加一个变体。类型系统强制执行穷举处理。回调架构将把此逻辑分散到数十个文件中，回调之间的交互将是隐式的而非在控制流中可见。

### 赌注 2：基于文件的内存超越数据库

第 11 章详细论证了，但架构意义延伸到内存之外。使用纯 Markdown 文件而非 SQLite、向量数据库或云服务的决定是透明性超越能力的赌注。数据库将支持更丰富的查询、更快的查找和事务保证。文件不提供这些。文件提供的是信任。

在 vim 中打开 `~/.claude/projects/myapp/memory/MEMORY.md` 并确切看到 agent 记住了他们什么的用户，与必须问 agent "你记得什么？"并希望答案完整的用户，与系统有根本不同的关系。基于文件的设计使 agent 的知识状态外部可观察，而不仅仅是自我报告的。这比查询性能更重要。LLM 驱动的召回系统通过检索智能弥补了存储简单性——从 manifest 中选择五个相关记忆的 Sonnet 副查询比嵌入相似度更精确，且需要零基础设施。

### 赌注 3：自描述工具超越中心编排器

Agent 框架通常提供工具注册表：你在中央配置中描述你的工具，框架将它们呈现给模型。Claude Code 的工具描述自己。每个 `Tool` 对象携带自己的名称、描述、输入 schema、prompt 贡献、并发安全标志和执行逻辑。工具系统的工作不是向模型描述工具——而是让工具描述自己。

这个赌注在可扩展性上得到回报。MCP 工具（第 15 章）通过实现相同接口成为一等公民。来自 MCP 服务器的工具和内置工具对模型是不可区分的。系统不需要单独的"MCP 工具适配器"层——包装产生标准 `Tool` 对象，从那时起，现有工具流水线处理它：权限检查、并发执行、结果预算、hook 拦截。

### 赌注 4：Fork Agent 用于缓存共享

第 9 章涵盖了 fork 机制：一个从其父的完整对话在其上下文窗口中开始的子 agent，共享父的 prompt cache。这不是便利优化——它是缓存共享模型值得 fork 生命周期管理复杂性的架构赌注。

替代方案——用对话摘要生成全新 agent——更简单但昂贵。每个新 agent 付费从头处理其上下文的全部成本。Fork agent 免费获得父的缓存前缀（输入 token 上 90% 折扣），使得为小任务生成 agent 变得经济：内存提取、代码审查、验证环节。后台内存提取 agent（第 11 章）在每个查询循环轮次后运行，其成本边缘化正是因为它共享父的缓存。没有基于 fork 的缓存共享，该 agent 将是禁止性昂贵的。

### 赌注 5：Hooks 超越 Plugins

大多数可扩展性系统使用 plugins——注册能力并在宿主进程内运行的代码。Claude Code 使用 hooks——在生命周期点运行并通过退出代码和 stdin/stdout 上的 JSON 通信的外部进程。

赌注是进程隔离值得开销。Plugin 可以崩溃宿主。Hook 崩溃自己的进程。Plugin 可以向宿主的堆泄漏内存。Hook 的内存随其进程消亡。Plugin 需要必须版本化和维护的 API 表面。Hook 需要 stdin、stdout 和退出代码——自 1971 年以来稳定的协议。

开销是真实的：每次 hook 调用生成进程花费内联回调不会花费的毫秒。内部回调的 -70% 快路径（第 12 章）表明系统知道此成本重要。但对于外部 hooks——用户脚本、团队 linter、企业策略服务器——隔离保证使系统更安全扩展。企业可以部署基于 hook 的策略强制执行而不担心畸形 hook 脚本会崩溃他们开发者的会话。

## 什么可转移，什么不可转移

不是 Claude Code 中的每个模式都通用化。一些是规模、资源或其他 agent 构建者可能不共享的特定约束的结果。

### 可转移到任何 Agent 的模式

**Generator 循环模式。** 任何需要流式处理响应、处理工具调用和管理多种终端状态的 agent 受益于使循环显式而非隐藏在回调之后。编码了循环为何停止的可辨识联合返回类型是消除整类"agent 为什么停止？"调试会话的模式。

**基于文件的内存与 LLM 召回。** 具体实现细节是 Claude Code 的，但原则——简单存储结合智能检索——适用于任何需要跨会话持久化知识的 agent。四类型分类法（user、feedback、project、reference）和可推导性测试（"这可以从当前项目状态重新推导吗？"）是可重用的设计启发法。

**远程执行的非对称读写通道。** 当读取是高频流而写入是低频 RPC 时，分离它们无论具体传输协议如何都是正确的。

**用于搜索的位图预过滤器。** 任何搜索大型文件索引的 agent 受益于 26 位字母位图作为预过滤器。每条目 4 字节，每个候选一次整数比较——成本效益比显著。

**Prompt cache 稳定性作为架构关注点。** 如果你的 agent 使用带 prompt caching 的 API，以稳定内容在前易变内容在后构建 prompt 不是优化——它是决定你成本结构的架构决策。

### Claude Code 规模特有的模式

**Fork 的终端渲染器。** Claude Code fork 了 Ink 并用紧凑的类型化数组、基于池的 interning 和单元格级 diff 重新实现了渲染流水线，因为它需要终端中的 60fps 流式输出。大多数 agent 渲染到 web 接口或简单日志输出。工程投资仅在终端渲染是你的主要 UI 且你以高频流式输出时才有意义。

**50+ 启动分析检查点。** 当你有数十万用户且 0.5% 采样产生统计显著数据时才有意义。对更小的 agent，更简单的计时机 suffices。

**八种 MCP 传输类型。** Claude Code 支持 stdio、SSE、HTTP、WebSocket、SDK、两种 IDE 变体和 Claude.ai 代理，因为必须与每种部署拓扑集成。大多数 agent 需要 stdio 和 HTTP。

**Hooks 快照安全模型。** 在启动时冻结 hook 配置且永不隐式重新读取是针对特定威胁的防御：在用户接受信任对话框后恶意仓库代码修改 hooks。当你的 agent 运行在带有不受信任 `.claude/` 配置的任意仓库中时，这很重要。仅在受信任环境中运行的 agent 可以使用更简单的 hook 管理。

## 复杂性的代价

近两千个文件。这买到了什么，又花费了什么？

文件计数作为复杂性指标是误导的。它的大部分是测试基础设施、类型定义、配置 schema 和 fork 的 Ink 渲染器。实际行为复杂性集中在少量高密度文件中：`query.ts`（1,700 行，agent 循环）、`hooks.ts`（4,900 行，生命周期拦截系统）、`REPL.tsx`（5,000 行，交互编排器）和内存系统的 prompt 构建函数。

复杂性来自三个来源，每个有不同的特点：

**协议多样性。** 支持五种终端键盘协议、八种 MCP 传输类型、四种远程执行拓扑和七个配置范围本质上是复杂的。每个额外协议是对代码库的线性添加，不是指数——但总和是大的。这种复杂性在 Brooksian 意义上是偶然的：它来自环境（终端碎片化、MCP 传输演化、远程部署拓扑），而非被解决的问题。

**性能优化。** 基于池的渲染、位图搜索预过滤器、sticky cache latch 和推测工具执行每个以可衡量的性能增益换取复杂性。这种复杂性由测量证明——每个优化都之前都有识别瓶颈的分析数据。风险是优化累积并以使热路径更难修改的方式交互。

**行为调优。** 内存系统的 prompt 指令、过期警告、验证协议、"忽略内存"反模式指令——这些不是代码复杂性。它们是 prompt 复杂性，携带不同的维护负担。当模型行为在版本之间改变时，通过 evals 仔细调优的 prompt 指令可能需要重调。Eval 基础设施（整个代码库中作为案例号和 eval 分数的引用）是防止回归的防御，但需要持续投资。

此系统的维护负担是显著的。阅读代码库的新工程师必须理解不仅是代码路径，还有激发特定 prompt 措辞的 eval 结果、激发特定安全检查的生产事件，以及激发特定优化的性能概况。代码注释是彻底的——许多包括 eval 案例号和前后测量——但近两千文件中的彻底注释本身就是阅读负担。

## Agentic 系统的发展方向

从 Claude Code 中的模式可见四个趋势，它们指向该领域走向何方。

### MCP 作为通用协议

第 15 章将 Claude Code 描述为最完整的 MCP 客户端之一。意义不是 Claude Code 的实现——而是 MCP 存在本身。用于工具发现和调用的标准化协议意味着为一个 agent 构建的工具与任何 agent 一起工作。生态系统效应明显：一个为 Postgres 构建的 MCP 服务器，一旦建成，服务每个说 MCP 的 agent。开发者在工具集成上的投资是可移植的。

对 agent 构建者的含义：如果你在定义自定义工具协议，你可能在犯错误。MCP 足够好，它在变得更好，标准协议的生态系统优势随时间累积。构建 MCP 客户端，贡献给规范，让协议通过社区反馈演进。

### 多 Agent 协调

Claude Code 的子 agent 系统（第 8 章）、task 协调（第 10 章）和 fork 机制（第 9 章）是多 agent 模式的早期实现。它们解决具体问题——缓存共享、并行探索、结构化验证——但也揭示了根本挑战：协调开销。

Agent 之间的每条消息消耗 token。每个 fork 共享缓存但添加了父最终必须协调的对话分支。Task 系统的状态机（queued、running、completed、failed、cancelled）是添加复杂性而不添加能力的协调机制。随着 agent 变得更有能力，压力将从"如何协调多个 agent？"转向"如何使一个 agent 足够有能力以至于协调是不必要的？"

当前证据表明两种方法将共存。简单任务将使用单个 agent。复杂任务将使用协调的多 agent 系统。工程挑战是使协调开销足够低，以至于交叉点有利于真正并行工作的多 agent，而不仅仅对于复杂的任务。

### 持久内存

Claude Code 的内存系统是持久 agent 内存的版本 1。基于文件的设计、四类型分类法、LLM 驱动的召回、过期系统和用于长时间运行会话的 KAIROS 模式都是将显著演进的问题的第一代解决方案。

未来内存系统可能会添加结构化检索（当前系统检索整个文件；未来系统可能检索特定事实）、跨项目迁移学习（适用于任何地方的用户偏好，不适用的项目约定）和协作内存（第 11 章的团队内存是第一步，但同步、冲突解决和访问控制是最小的）。

开放问题是基于文件的方法是否可扩展。在每个项目 200 个记忆时，它工作。在每个项目 2,000 个记忆时，Sonnet 副查询的 manifest 变得太大，合并变得太昂贵，索引超过其上限。随着使用增长，文件超越数据库的架构赌注将面临其最艰难的考验。

### 自主操作

KAIROS 模式、后台内存提取 agent、auto-dream 合并、推测工具执行——这些都是朝自主操作的步骤。Agent 在未被要求的情况下做有用的工作：它记住你忘记告诉它要记住的东西，在你睡觉时合并自己的知识，在当前响应完成之前开始执行下一个工具。

轨迹是清晰的。未来 agent 将更少反应性而更多主动性。它们将注意到用户未描述的模式，建议用户未请求的纠正，并在没有显式 `/remember` 命令的情况下维护自己的知识。Claude Code 的内存系统，带有其后台提取安全网和其 prompt 工程的"保存什么"启发法，是这个未来的原型。

约束是信任。自主操作要求用户信任 agent 在无人值守时会做正确的事。基于文件的内存、可观察的 hook 系统、过期警告、权限对话框——所有这些存在是因为信任必须被赢得，而非假定。通往更自主 agent 的路径经过更透明的 agent。

## 结语

十七章。六个核心抽象。中心的 generator 循环，工具向外延伸，内存穿越时间向回延伸，hooks 守卫边界，渲染引擎将其全部转换为屏幕上的字符，MCP 将其连接到代码库之外的世界。

Claude Code 中最深的模式不是任何单一技术。它是将复杂性推向边界的反复决策。渲染系统将复杂性推到池和 diff——管道内部，一切都是整数比较。输入系统将复杂性推到 tokenizer 和 keybinding resolver——处理器内部，一切都是类型化动作。内存系统将复杂性推到写入协议和召回选择器——对话内部，一切都是上下文。Agent 循环将复杂性推到终端状态和工具系统——循环内部，只是：流式、收集、执行、追加、重复。

每个边界吸收混乱并输出秩序。原始字节变成 `ParsedKey`。Markdown 文件变成召回的記憶。MCP JSON-RPC 变成 `Tool` 对象。Hook 退出代码变成权限决策。在每个边界的一侧，世界是混乱的——五种键盘协议、脆弱的 OAuth 服务器、过时的记忆、不受信任的仓库 hooks。在另一侧，世界是类型化、有界和穷举处理的。

如果你在构建 agentic 系统，这是可转移的教训。不是具体技术——你可能不需要基于池的渲染或 KAIROS 模式或八种 MCP 传输。而是原则：定义你的边界，在那里吸收复杂性，并保持它们之间的一切干净。边界是工程困难的地方。内部是工程愉快的地方。为愉快的内部设计，在边界投资你的复杂性预算。

源码是开放的。螃蟹的钳子里有地图。去读它。
