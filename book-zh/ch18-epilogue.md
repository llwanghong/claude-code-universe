# 第 18 章：我们学到了什么

## 五个架构赌注

Claude Code 不是唯一的 agentic 系统。它不是第一个。但它在五个架构赌注上区别于 agent 框架的景观，经过近两千个文件和十七章，这些赌注值得审视。

### 赌注 1：Generator Loop 超过 Callbacks

大多数 agent 框架给你一个管道：定义工具，注册处理器，让框架编排。开发者写回调。框架决定何时调用它们。

Claude Code 做相反的事。`query()` 函数是一个 async generator——开发者拥有循环。模型流式返回响应，generator 产出工具调用，调用者执行它们，追加结果，generator 循环。有一个函数，一个数据流，一个所有交互都经过的地方。10 个终端状态和 7 个 continuation 状态编码了每种可能的结果。循环就是系统。

赌注是，一个单一的 generator 函数，即使长到 1,700 行，也会比分布式回调图更容易理解。在研究了源码之后，赌注获得了回报。当你想理解为什么会话结束时，你看一个函数。当你想添加一个新的终端状态时，你向一个可辨识联合类型添加一个变体。类型系统强制穷举处理。回调架构会把这个逻辑分散到几十个文件中。

### 赌注 2：基于文件的内存超过数据库

第 11 章在细节上阐述了理由，但架构意义超越了内存。决定使用纯 Markdown 文件而不是 SQLite、向量数据库或云服务是透明性优先于能力的赌注。数据库会支持更丰富的查询、更快的查找和事务保证。文件不提供这些。文件提供的是信任。

在 vim 中打开 `~/.claude/projects/myapp/memory/MEMORY.md` 并确切看到 agent 记住了关于他们的什么的用户，与必须问 agent "你记得什么？" 并希望答案完整的用户具有根本不同的关系。基于文件的设计使 agent 的知识状态外部可观察而不仅仅是自我报告的。这比查询性能更重要。LLM 驱动的召回系统以检索智能补偿了存储简单性——从清单中选择五个相关记忆的 Sonnet 侧查询比嵌入相似度更精确，且需要零基础设施。

### 赌注 3：自我描述的工具超过中央编排器

Agent 框架通常提供一个工具注册表：你在中央配置中描述你的工具，框架将它们呈现给模型。Claude Code 的工具描述自己。每个 `Tool` 对象携带自己的名称、描述、输入 schema、prompt 贡献、并发安全标志和执行逻辑。工具系统的工作不是向模型描述工具——它是让工具描述自己。

这个赌注在可扩展性上获得回报。MCP 工具（第 15 章）通过实现相同的接口成为一等公民。来自 MCP 服务器的工具和内置工具对于模型来说是不可区分的。系统不需要单独的 "MCP 工具适配器" 层——包装产生一个标准的 `Tool` 对象，从那一刻起，现有的工具管道处理它。

### 赌注 4：Fork Agent 用于 Cache 共享

第 9 章涵盖了 fork 机制：一个从父级的完整对话开始的子 agent，共享父级的 prompt cache。这不是便利优化——它是一个架构赌注，即 cache 共享模型值得 fork 生命周期管理的复杂性。

替代方案——用一个对话摘要生成一个新 agent——更简单但昂贵。每个新 agent 支付从头处理其上下文的全部成本。一个 forked agent 免费获得父级的缓存前缀（输入 token 约 90% 的折扣），使得为小任务生成 agent——memory 提取、代码审查、验证通行——在经济上可行。没有基于 fork 的 cache 共享，后台 memory 提取 agent 的成本将是禁止性的。

### 赌注 5：Hooks 超过 Plugins

大多数可扩展性系统使用插件——注册能力并在主机进程内运行的代码。Claude Code 使用 hooks——在生命周期点运行并通过退出码和 stdin/stdout 上的 JSON 进行通信的外部进程。

赌注是进程隔离值得开销。插件可以使主机崩溃。Hook 崩溃自己的进程。插件可能将内存泄漏到主机的堆中。Hook 的内存随其进程而消亡。插件需要一个必须版本化和维护的 API 表面。Hook 需要 stdin、stdout 和一个退出码——自 1971 年以来稳定的协议。

开销是真实存在的：每次 hook 调用生成一个进程花费几毫秒，这是进程内回调不会花费的。内部回调的 -70% 快速路径（第 12 章）表明系统知道这个成本很重要。但对于外部 hooks——用户脚本、团队 linter、企业策略服务器——隔离保证使系统更安全地扩展。

## 什么可转移，什么不可转移

并非 Claude Code 中的每个模式都能泛化。有些是规模、资源或特定约束的后果。

### 可转移到任何 Agent 的模式

**Generator loop 模式。** 任何需要流式响应、处理工具调用和管理多个终端状态的 agent 都受益于使循环显式而不是隐藏在回调之后。编码了循环为何停止的可辨识联合类型返回类型消除了整类的 "agent 为什么停止？" 调试会话。

**基于文件的内存带 LLM 召回。** 具体实现细节是 Claude Code 的，但原则——简单存储结合智能检索——适用于任何需要跨会话持久化知识的 agent。四种类型分类法和可推导性测试是可重用的设计启发式。

**用于远程执行的非对称读写通道。** 当读取是高频流且写入是低频 RPC 时，分离它们是正确的，不管具体的传输协议是什么。

**用于搜索的位图预过滤器。** 任何搜索大文件索引的 agent 都受益于 26 位字母位图作为预过滤器。四个字节每条目，一次整数比较每候选。成本效益比异常出色。

**Prompt cache 稳定性作为架构关注点。** 如果你的 agent 使用带有 prompt caching 的 API，用稳定内容在前和易变内容在后来构建 prompt 不是一个优化——它是决定你成本结构的架构决策。

### Claude Code 规模特有的模式

**Forked 终端渲染器。** 当终端渲染是你的主要 UI 并且你在高频率流式传输时值得工程投资。

**50+ 个启动剖析检查点。** 当你有数十万用户且 0.5% 采样产生统计显著数据时有意义。

**八种 MCP 传输类型。** 大多数 agent 需要 stdio 和 HTTP。

**Hooks 快照安全模型。** 当你的 agent 在具有不受信任的 `.claude/` 配置的任意仓库中运行时，这很重要。

## 复杂性的代价

近两千个文件。这买到了什么，又花费了什么？

文件计数作为复杂性指标是误导性的。其中很多是测试基础设施、类型定义、配置 schema 和 forked Ink 渲染器。实际的行为复杂性集中在少量的高密度文件中：`query.ts`（1,700 行，agent loop）、`hooks.ts`（4,900 行，生命周期拦截系统）、`REPL.tsx`（5,000 行，交互式编排器）以及 memory 系统的 prompt 构建函数。

复杂性来自三个来源：**协议多样性**（支持五种终端键盘协议、八种 MCP 传输类型、四种远程执行拓扑和七个配置范围本质上是复杂的）、**性能优化**（pool-based 渲染、位图搜索预过滤器、sticky cache latches 和推测性工具执行各自以可衡量的性能增益换取复杂性）、以及**行为调优**（memory 系统的 prompt 指令、过时警告、验证协议不是代码复杂性——它们是 prompt 复杂性，并带有不同的维护负担）。

该系统的维护负担是显著的。阅读代码库的新工程师必须理解不仅代码路径，还有促使特定 prompt 措辞的 eval 结果、促使特定安全检查的生产事件，以及促使特定优化的性能剖析文件。代码注释是详尽的——许多包含 eval 案例编号和前后测量——但近两千个文件中的详尽注释本身就是阅读负担。

## 一个可转移的教训

如果你正在构建一个 agentic 系统，这是可转移的教训。不是特定的技术——你可能不需要 pool-based 渲染或 KAIROS 模式或八种 MCP 传输。而是原则：**定义你的边界，在那里吸收复杂性，保持边界之间的东西干净。**

每个边界吸收混乱并导出秩序。原始字节变成 `ParsedKey`。Markdown 文件变成召回的 memory。MCP JSON-RPC 变成 `Tool` 对象。Hook 退出码变成权限决策。在每条边界的一侧，世界是混乱的——五种键盘协议、脆弱的 OAuth 服务器、过时的记忆、不受信任的仓库 hooks。在另一侧，世界是类型化的、有界限的、穷举处理的。

边界是工程困难的地方。内部是工程愉快的地方。为愉快的内部设计，将你的复杂性预算投资在边界。

源码是开放的。螃蟹的爪子里拿着地图。去读它。
