# 第 18 章：我们学到了什么

## 五个架构赌注

Claude Code 不是唯一的 agent 系统。但它在五个架构赌注上区别于 agent 框架的景观。经过近两千个文件和十七章，这些赌注值得审视。

### 赌注 1：Generator 循环超过回调

大多数 agent 框架给你一个流水线：定义工具，注册处理器，让框架编排。开发者写回调。框架决定何时调用它们。

Claude Code 做相反的事。`query()` 函数是一个 async generator——开发者拥有循环。模型流式返回响应，generator 产出工具调用，调用者执行它们，追加结果，generator 循环。有一个函数，一个数据流，一个所有交互都经过的地方。

这个赌注是，一个单一的 generator 函数，即使长到 1,700 行，也比分布式回调图更容易理解。当你想理解为什么会话结束时，你看一个函数。类型系统强制穷举处理。回调架构会把这个逻辑分散到几十个文件中。

### 赌注 2：基于文件的内存超过数据库

决定使用纯 Markdown 文件而不是 SQLite、向量数据库或云服务是关于透明性胜过能力的赌注。数据库会支持更丰富的查询、更快的查找和事务保证。文件不提供这些。文件提供的是信任。

打开 `~/.claude/projects/myapp/memory/MEMORY.md` 并在 vim 中看到 agent 记得关于你的什么，这与必须问 agent "你记得什么？"并希望答案完整有根本不同。基于文件的设计使 agent 的知识状态外部可观察而不仅仅是自我报告的。这比查询性能更重要。

### 赌注 3：自我描述的工具超过中央编排器

Agent 框架通常提供工具注册表：你在中央配置中描述你的工具，框架将它们呈现给模型。Claude Code 的工具描述自己。每个 `Tool` 对象携带自己的名称、描述、输入 schema、prompt 贡献、并发安全标志和执行逻辑。

这个赌注在可扩展性上获得回报。MCP 工具通过实现相同的接口成为一等公民。来自 MCP 服务器的工具和内置工具对于模型来说是不可区分的。

### 赌注 4：Fork Agent 用于 Cache 共享

第 9 章介绍了 fork 机制：一个从父级完整对话开始的子 agent，共享父级的 prompt cache。这不是便利优化——它是一个架构赌注，cache 共享模型值得 fork 生命周期管理的复杂性。

一个 forked agent 继承父级的缓存前缀以获得约 90% 的输入 token 折扣，使得为小任务（如内存提取、代码审查、验证）启动 agent 变得经济可行。

### 赌注 5：Hooks 超过插件

大多数可扩展性系统使用插件——注册能力并在主机进程内运行的代码。Claude Code 使用 hooks——在生命周期点运行并通过退出码和 stdin/stdout 上的 JSON 进行通信的外部进程。

赌注是进程隔离值得开销。插件可以使主机崩溃。Hook 崩溃的是自己的进程。进程隔离的代价是每次调用的毫秒数开销。但对于外部 hooks——用户脚本、团队 linter、企业策略服务器——隔离保证使系统更安全地扩展。

## 哪些模式可转移，哪些不能

### 可转移到任何 Agent 的模式

- **Generator 循环模式。** 任何需要流式响应、处理工具调用和管理多个终端状态的 agent 都受益于此。
- **基于文件的内存 + LLM 召回。** 简单存储 + 智能检索的原则适用于任何需要跨会话持久化知识的 agent。
- **非对称读写通道用于远程执行。** 当读取是高频流且写入是低频 RPC 时，分离它们。
- **位图预过滤器用于搜索。** 四个字节 per 条目，一个整数比较 per 候选。成本效益比惊人。
- **Prompt cache 稳定性作为架构关注点。** 稳定内容在前，易变内容在后——不是优化，这是架构决策。

### Claude Code 规模特有的模式

- **Fork 的终端渲染器。** 只有终端渲染是你的主要 UI 并且你在高频率流式传输时，工程投资才有意义。
- **50+ 启动性能检查点。** 当你有数十万用户且 0.5% 采样产生统计显著数据时有意义。
- **八种 MCP 传输类型。** 大多数 agent 需要 stdio 和 HTTP。
- **Hooks 快照安全模型。** 当你的 agent 在任意仓库中运行时才真正必要。

## Agent 系统的发展方向

1. **MCP 作为通用协议。** 为 Postgres 构建的 MCP 服务器服务于每个说 MCP 的 agent。如果你在定义自定义工具协议，你可能在犯错误。
2. **多 Agent 协调。** 简单任务使用单个 agent。复杂任务使用协调的多 agent 系统。工程挑战是使协调开销足够低。
3. **持久化内存。** 未来的内存系统将添加结构化检索和跨项目迁移学习。文件方法能否扩展到 2,000 个记忆是一个开放问题。
4. **自主操作。** 未来的 agent 将更少响应式，更多主动式——注意到用户未描述的模式、建议用户未要求的更正。

## 一个可转移的教训

如果你要构建一个 agent 系统，不是特定的技术——你可能不需要基于 pool 的渲染或 KAIROS 模式或八种 MCP 传输。但原则是：**定义你的边界，在边界处吸收复杂性，保持边界之间的东西干净。** 每个边界吸收混乱并导出秩序。原始字节变成 `ParsedKey`。Markdown 文件变成召回的记忆。MCP JSON-RPC 变成 `Tool` 对象。Hook 退出码变成权限决策。在每条边界的一侧，世界是混乱的。在另一侧，世界是类型化的、有界限的和穷举处理的。

源码是开放的。螃蟹的爪子里拿着地图。去读它。
