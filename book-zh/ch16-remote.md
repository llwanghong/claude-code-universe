# 第 16 章：远程控制与云端执行

## 超越本地主机

Claude Code 主要是一个本地 CLI。但远程执行允许 agent 在远程机器上运行，同时保持本地 UI。这对于云开发环境、企业部署策略以及本地机器不够强大的场景是必要的。

远程架构的核心洞察是读取和写入有不同的带宽需求。读取（远程 → 本地）是高频率流——每个 token 生成一个事件。写入（本地 → 远程）是低频 RPC——仅在用户输入或工具审批时。为每个方向使用相同的传输会为不需要的部分优化错误的东西。

## Bridge 协议

Bridge 连接本地 REPL 和远程 agent loop。两个版本：

**Bridge v1**（轮询）。本地 REPL 按间隔轮询远程端的状态变化。简单、可防火墙穿透、易于调试。高延迟——每个轮询间隔增加平均一半的延迟。

**Bridge v2**（事件驱动）。远程端通过 Server-Sent Events 推送事件（消息、工具调用、状态变化）到本地 REPL。较低延迟，较复杂。使用 SSE 进行单向流，HTTP POST 进行命令。读取方向和写入方向是不同的传输。

协议差异是透明的：REPL 在这两种情况下都看到相同的 `Message` 对象。Bridge 版本是一个部署选择，而不是一个架构选择——相同的 agent loop，相同的工具系统，相同的行为。

## Claude Code Runtime（CCR）

CCR 是远程执行环境。它在云端运行 agent loop，同时将事件流式传输回本地客户端：

- 本地 REPL 渲染 UI 并处理用户输入
- 远程运行时执行工具、调用模型、管理上下文
- Bridge 协议同步两者

CCR 配置在启动时指定。如果配置了远程运行时，本地 REPL 连接到它而不是生成本地 agent loop。如果连接失败，系统以警告回退到本地执行。

## 非对称通道

设计的核心洞察：读取和写入有不同的带宽。

- **读取**（远程 → 本地）：高频率流。每个 token 生成一个事件。每秒数百个事件。SSE 是正确选择——单向、可缓存、防火墙友好，使用标准 HTTP 基础设施。
- **写入**（本地 → 远程）：低频 RPC。仅在用户输入或工具审批时。每分钟可能几个事件。HTTP POST 是正确选择——标准、无处不在、易于通过代理。

为两个方向使用相同的协议（WebSocket 在两个方向上都是双向的）会为不需要的部分优化错误的东西。

## Upstream Proxy

对于企业客户，upstream 代理拦截 API 调用用于：速率限制和成本跟踪、审计日志、策略强制执行和自定义模型路由。

Proxy 在启动时通过环境变量和 CLI 标志配置。Agent loop 不知道 proxy——它看到的是标准的 Anthropic 客户端接口（第 4 章）。Proxy 是一个基础设施关注点，在 agent loop 之外处理。

## Apply This

**为读取和写入使用不同的通道。** 读取是高带宽流，写入是低频 RPC。为每个需求优化。SSE 用于流式事件，HTTP POST 用于命令。

**SSE 用于流式事件。** 比 WebSocket 简单，通过所有代理和防火墙工作，使用标准 HTTP 基础设施，原生自动重连。不需要双向通信时不要引入。

**Bridge 模式分离呈现和计算。** 本地渲染，远程执行。UI 保持响应，计算获得云资源。Bridge 协议是这两个世界之间的接缝——确保它足够薄以保持透明。

**审计对企业很重要。** 如果你在构建企业 agent 基础设施，从第一天起就构建审计——不只是遥测，而是完整的请求/响应日志。Upstream proxy 是正确的地方。Agent loop 不应该知道审计。

**Upstream proxy 处理横切关注点。** 速率限制、成本跟踪、策略强制执行属于 proxy 层，而不属于 agent loop。Agent loop 看到的是一个干净的 API 客户端。Proxy 在基础设施层强制执行组织策略。
