# 第 14 章：输入与交互

## 终端键盘输入之坑

终端键盘输入是一团糟。不同的终端仿真器对相同的物理按键发送不同的字节序列。Escape 是 `\x1b`。但 Option+Left 可能是 `\x1b[1;3D`，或者 `\x1b\x1b[D`，或者完全是别的什么，取决于终端。

Claude Code 必须干净地处理这个问题，因为开发者在终端中花费数小时。坏的键盘处理会立即让用户流失。

## 键解析管道

```
原始字节 → ParsedKey → 键绑定解析器 → 动作分发 → 处理程序
```

**ParsedKey** 是规范表示：`{ key: 'left', ctrl: true, shift: false, option: true }`。解析器从五个不同的终端协议中吸收所有原始字节变化，并输出一个统一的键对象。

**键绑定解析器**将规范键映射为用户可配置的动作。它支持：
- 简单绑定：`ctrl+n` → `nextItem`
- Chord 绑定：`ctrl+k` 然后 `ctrl+f` → `formatCode`
- 模式特定绑定：在 vim 的普通模式与插入模式下，相同的按键做不同的事情
- 插件提供的绑定

**动作分发**将动作名称映射到处理程序函数。处理程序更新 REPL 状态（移动光标、提交输入、切换模式）并触发 UI 重新渲染。

## Vim 模式

Claude Code 支持 vim 风格的键绑定进行文本编辑。输入区域有一个模态编辑器：
- 普通模式：移动、删除、yank、indent 的键绑定
- 插入模式：文本输入

Vim 模式使用与主键绑定系统相同的 `ParsedKey` → 动作管道，但带有一组不同的动作和处理程序。

## Apply This

1. **尽早将原始输入规范化为规范形式。** 在边界处解析一次字节 → `ParsedKey`。让后续的所有内容处理干净的类型化动作。
2. **支持 chords 和多种模式。** 单键绑定对于专业工具是不够的。构建一个知道模式和 chords 的键绑定系统。
3. **使键绑定可配置。** 用户对键绑定有强烈的偏好。将它们存储在配置文件中，而不是硬编码。
4. **Vim 模式不值得全部从头实现。** 但子集——移动、删除、yank、基本模式——在终端时间方面有很高的投资回报率。
5. **边界吸收混乱。** 五个终端协议在管道的开头被处理。其余的一切只看到 `ParsedKey`。在边界处吸收复杂性。
