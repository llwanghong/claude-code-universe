# 第 3 章：状态 — 双层架构

## 基础设施不响应

Claude Code 在两个层面上维护状态。分离是有意的：基础设施状态不经常变化也不触发重新渲染。UI 状态经常变化且必须触发。

## Bootstrap 单例（STATE）

一个可变的全局单例（`STATE`）持有约 80 个字段的会话级基础设施：

- 工作目录、会话 ID、模型配置
- 成本追踪计数器
- 遥测缓冲区
- Feature flags
- API 密钥和认证状态

它在启动时设置一次并直接修改——没有响应性。没有订阅。没有变更通知。当 `cost-tracker.ts` 递增成本计数器时，没有任何 UI 组件收到通知。状态只是更新。

## AppState Store

一个最小化的响应式 store（34 行，Zustand 风格的）驱动 UI：

- 消息数组
- 输入模式（普通、权限提示、计划编辑）
- 工具批准对话框
- 进度指示器
- Spinner 状态

只有 UI 需要的状态是可响应的。其他一切都是直接的。

## Sticky Latches

一个微妙但重要的模式：某些决策必须对整个会话保持固定。

一旦会话开始，发送到 API 的 beta headers 永远不会改变。即使新的 beta 功能在会话期间变得可用，headers 也会保持锁定。类似地，feature flag 评估在启动时快照，永远不会重新评估。

为什么：改变 beta headers 会无效整个 prompt cache。模型看到不同的 headers，将请求视为不同，并重新处理整个上下文。为了缓存稳定性，sticky latches 牺牲了灵活性。

## 成本追踪

成本追踪器维护：
- 按模型类型（输入/输出/cache 创建/cache 读取）的 token 计数
- 累计美元成本
- 按查询源的每轮成本

由于 API 响应在流的过程中不包含成本信息，追踪器估算。当实际成本可用时（在流后），估算被修正。

## Apply This

1. **分离基础设施状态和 UI 状态。** 基础设施不会频繁变化也不需要触发重新渲染。不要为它支付响应式开销。
2. **Sticky latches 有利于缓存稳定性。** 一旦为某事物选择了一个值，就坚持整个会话。改变会无效化缓存。
3. **在启动时快照配置。** 在会话期间重新读取配置是变化和不一致性的来源。快照一次，使用到结束。
4. **估算然后修正。** 当实际数据有延迟时，先估算。当实际数据到达时修正。不要让完美成为良好的敌人。
5. **成本追踪不是可选的。** 如果你正在运行一个使用 API 的 agent，每个用户交互的 token 计数和美元成本不是遥测——它们是产品。
