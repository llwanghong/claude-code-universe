# 第 17 章：性能 — 每毫秒和每个 Token 都很重要

## 为什么性能是架构性的

性能不仅仅是让事情变快。它是关于在性能约束下做出不同的架构选择。Claude Code 做出的每一个架构决策——async generator loop、packed array 渲染、sticky latches、推测性工具执行——都是由性能驱动的。

对数十万用户来说，100ms 的延迟是显著的。单个 token 开销了 90% 是季度间六位数的差异。

## 启动性能

第 2 章在细节上介绍了启动。从性能角度来看，关键数字：

- **模块评估**：~138ms（最长的单阶段）
- **并行 I/O**：与模块评估重叠（MDM、keychain）
- **总启动预算**：300ms

保持低于 300ms 的策略：
- 动态 import（推迟模块评估直到需要）
- 模块级 Promise（并行 I/O 与 import 重叠）
- 快速路径分发（在加载全系统之前退出）

## Context Window 优化

Context window 是代理最稀缺的资源。优化：

- **Slot reservation**：默认 8K 输出；在需要时升级到 64K。在 99% 的请求中节省上下文。
- **Prompt cache**：稳定内容在前，易变内容在后。缓存前缀的 90% 折扣。
- **上下文压缩**：5 层压缩流水线保持对话在限制之下。

## Prompt Cache 策略

Prompt caching 将成本降低 90%。优化它的策略：

- **Sticky latches**：Beta headers 永远不变
- **工具定义顺序**：在重启之间保持稳定
- **System prompt 确定论**：对相同输入，system prompt 生成必须返回相同输出
- **动态内容分离**：易变内容（对话历史）放在最后

## 渲染性能

- **Packed arrays**：每个单元格 4 字节，不是 1 个 JS 对象
- **Cell-level diffing**：每帧只输出更改的单元格
- **Pool-based interning**：样式对象被 intern 而不是分配

## 搜索性能

对于大代码库中的文件搜索：
- 内置的 ripgrep（通过 vendor/ripgrep）
- 26 位字母位图作为预过滤器：检查文件是否包含搜索字符串中的所有字母，然后才启动 rg。四个字节 per 文件，一次整数比较。成本效益比惊人。

## 性能剖析

代码库包含 50+ 个性能检查点。这些是由环境变量启用的计时检查点，以在所有启动和查询循环阶段的随机用户子集上收集计时数据。数据被采样（0.5%）以避免性能开销。

## Apply This

1. **测量，不要假设。** 50+ 个检查点不仅仅是调试辅助——它们是你关于"什么慢"的数据源和不进行没有数据的优化的约束。
2. **Prompt cache 稳定性是架构性的。** 不是优化。System prompt 和工具定义中的字节顺序决定了你的成本结构。
3. **Slot reservation：预留少，按需升级。** 默认预留 8K 输出 tokens。仅在模型命中上限时升级到 64K。
4. **Packed arrays 用于频繁变化的数据。** 当 60fps 比较数千个单元格时，typed arrays > 对象数组。
5. **位图预过滤器是免费的。** 四个字节 per 条目，一个整数比较。如果它消除 80% 的昂贵搜索，它就值得。
