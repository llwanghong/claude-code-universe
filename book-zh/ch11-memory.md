# 第 11 章：记忆 — 跨会话学习

## 无状态的困境

Claude Code 在一个会话中知道很多。它可以读取代码库、执行命令、推理输出。但当你退出并重新启动时，所有这些上下文都会消失。Agent 关于你的项目约定、调试历史和架构决策的知识都被遗忘了。

Memory 系统修复了这个问题，使用简单的基于文件的存储和 LLM 驱动的检索。

---

## 三层

Memory 存在于三个层级：

**项目级**（仓库中的 `CLAUDE.md` 文件）。每个仓库可以包含一个或多个记忆文件。这些对于在该代码库上工作的每个人都是共享的。它们通常存储代码库约定、构建说明、架构决策和常见陷阱。

**用户级**（`~/.claude/MEMORY.md`）。跨项目适用的个人偏好和知识。首选的编码风格、常用的 API 密钥名称、个人工具偏好。

**团队级**（通过符号链接共享的 `CLAUDE.md` 文件）。组织范围的知识，如安全策略、合规要求和共享基础设施文档。

---

## 四种记忆类型

系统对记忆进行分类：

| 类型 | 存储内容 | 示例 |
|------|---------|------|
| `user` | 用户是谁（角色、专长、偏好） | "偏好函数式风格" |
| `feedback` | 用户对 agent 应该如何工作的指导 | "不要直接修改 config.ts" |
| `project` | 无法从代码或 git 历史推断的进行中的工作 | "正在将认证系统迁移到 JWT" |
| `reference` | 外部资源指针 | "API 文档在..." |

类型决定记忆如何被召回、何时显示以及如何处理过时。

---

## 可推导性测试

决定是否存储某事物：**"这可以从当前项目状态重新推导出来吗？"** 如果可以——不要存储。记忆文件应该只保存*不能*从代码库、git 历史或项目结构重新推导的东西。

代码约定在代码中可见。函数签名在代码中可见。迁移计划、用户偏好和口头约定都是不可见的。保存不可见的东西。

---

## 写入路径

**显式**：用户输入"记住：我们正在迁移到 TypeScript strict mode"或使用 `/remember` 命令。提示被解析，frontmatter 字段被提取，记忆文件被保存。

**隐式**：后台 memory 提取 agent（在每次查询循环轮次后运行，由 `KAIROS` feature flag 门控）扫描最近的对话，寻找可能在以后有用的持久化信息。它应用可推导性测试并有选择地保存。Agent 被提示宁可漏掉也不过度捕获——false negative 无害，true positive 会使记忆存储膨胀。

每个记忆是一个文件：`~/.claude/projects/<project>/memory/<slug>.md`。它有 YAML frontmatter（名称、描述、元数据）和 Markdown body。

---

## 召回路径

在会话开始时，系统扫描所有记忆文件，解析 frontmatter，建立清单。清单包含每个记忆的名称、描述和元数据——不包含 body。

一个轻量级的 Sonnet 侧查询接收清单和当前对话上下文。它选择最多 5 个与当前对话最相关的记忆。只有选中的记忆被注入到 system prompt 中。

Sonnet 召回胜过嵌入搜索因为它理解*意图*。一个关于"error handling patterns"的记忆应该匹配"如何传播异常？"和"try-catch 看起来不对"。嵌入向量是不精确的——Sonnet 不是。

---

## 过时系统

记忆会过时。一个在周一准确的约定在周三可能已经改变。System 通过比较最后修改时间戳和相关代码的 git 历史来检测潜在过时的记忆。当检测到可能过时的记忆时，agent 会通知用户并询问该记忆是否仍然适用。

过时不是失败——它是被预期的。System 承认记忆可能过时需要清理，并且让用户明确意识到这种可能性。

---

## 为什么基于文件而不是数据库

基于文件的内存是五大约赌注之一（第 18 章）。用纯文本文件替代 SQLite 或向量数据库是一种透明性优先于能力的赌注。用户可以在任何编辑器中打开 `MEMORY.md`。Agent 的知识状态是外部可观察的而不是自我报告的。LLM 驱动的召回比嵌入搜索更聪明。

代价：没有向量查询、没有全文索引、没有事务保证。但在 200 个记忆的规模下，简单性在实践中胜出。

---

## Apply This

**简单存储 + 智能检索。** 将记忆存储为带有 frontmatter 元数据的纯文本文件。在检索时使用 LLM 选择相关记忆。不要过度设计存储。LLM 驱动的召回在不需要基础设施的情况下胜过嵌入相似度。

**可推导性测试。** 如果某事物可以从当前项目状态重新推导——不要存储。只保存代码和 git 历史中不可见的东西。节省上下文，避免冗余。

**按类型分类记忆。** User、feedback、project、reference。每种类型有不同的生命周期和召回策略。类型决定过时如何被检测以及用户何时被通知。

**过时不是失败——它是被预期的。** 在相关对话期间警告用户可能存在过时的记忆。让他们决定清理。不要默默地使用过期信息。信任要求外部可观察性。

**外部可观察性建立信任。** 用户应该能够在不经过 agent 的情况下看到 agent 记住了什么。基于文件的记忆使知识状态可审计。信任是被赢得的——不是声明的。
