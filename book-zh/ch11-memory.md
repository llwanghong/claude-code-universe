# 第 11 章：记忆 — 跨会话学习

## 没有记忆的 Agent 永远重复同样的错误

Claude Code 的记忆系统回答了"agent 如何跨会话学习？"三层：项目级（仓库中的 `CLAUDE.md`）、用户级（`~/.claude/MEMORY.md`）和团队级（通过符号链接共享的 `CLAUDE.md`）。

## 四种记忆类型

| 类型 | 存储内容 | 示例 |
|------|---------|------|
| `user` | 用户是谁（角色、专长、偏好） | "你偏好函数式风格" |
| `feedback` | 用户对你应该如何工作的指导 | "用这个 API endpoint" |
| `project` | 从代码或 git 历史无法推断的进行中的工作 | "我们正在迁移到 TypeScript" |
| `reference` | 外部资源指针 | "API 文档在…" |

## 可推导性测试

决定是否存储某事物："这可以从当前项目状态重新推导出来吗？"如果可以——不要存储。记忆文件应该只保存*不能*从代码库、git 历史或项目结构重新推导的东西。

## 记忆如何工作

**写入路径：** 当用户说 "记住 X"（或 `/remember` 命令），或后台记忆提取 agent 注意到有值得保存的东西时，它写入一个带有 YAML frontmatter 的 Markdown 文件（名称、描述、元数据）。每个记忆一个文件。

**召回路径：** 在会话开始时，系统扫描所有记忆文件，解析 frontmatter，构建清单。一个轻量的 Sonnet 侧面查询选择 5 个与当前对话最相关的记忆。这些在 system prompt 中注入。

**过时系统：** 记忆可能过时。系统在相关对话期间警告过时的记忆，告知用户哪些记忆可能不再适用及其原因。

## 为什么基于文件而不是数据库

- 用户可以用任何编辑器打开记忆
- Git diff 显示记忆如何随时间变化
- LLM 驱动的召回胜过嵌入相似度——Sonnet 理解*
- 零基础设施——没有向量数据库，没有 API 密钥

代价：查询功能有限（没有向量搜索，没有全文索引）。但在 200 个记忆的规模下，这已经足够好了。

## Apply This

1. **简单存储 + 智能检索。** 将记忆存储为纯 Markdown 文件。用 LLM 在检索时选择相关记忆。
2. **可推导性测试。** 如果某事物可以从当前状态重新推导出来——不要存储。只保存无法重新推导的东西。
3. **按类型分类记忆。** `user`、`feedback`、`project`、`reference`。每种类型有不同的生命周期和召回策略。
4. **过时不是故障——它是被预期的。** 记忆会过时。构建过时检测和警告。不要让用户自己发现。
5. **外部可观察性建立信任。** 用户应该不通过询问 agent 就能看到 agent 记得什么。基于文件的记忆使这成为可能。
