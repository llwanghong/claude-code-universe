# S01：Bootstrap 与状态管理

> 对应：书 ch02 + ch03 | 源码：bootstrap/state.ts, main.tsx, cli.tsx

## 为什么前端工程师要关心这个

Claude Code 的启动流水线是「如何在 300ms 内启动一个复杂系统」的实战案例。前端应用面临同样的问题——你的 React SPA 启动需要加载配置、读取 cookie、验证认证、初始化路由，每多 100ms 用户流失率就上升。

## 核心模式

### 1. 漏斗式初始化

```
Phase 0: 检查是否真的需要全量启动（--version？提前退出）
Phase 1: 模块级 I/O 并行（import 副作用触发子进程）
Phase 2: 解析 + 信任边界（配置、安全决策）
Phase 3: 并行注册（命令、hooks、插件、MCP 全部并行）
Phase 4: 选择启动路径 → 首次渲染
```

**前端对应**：SPA 启动 → 检查路由是否需要登录 → 并行加载用户数据 + i18n + 主题 → 渲染

### 2. 双层状态分离

- **STATE 单例**（不可响应）：基础设施状态，变化不频繁，不需要触发重渲染（80 个字段）
- **AppState store**（响应式）：仅 UI 需要的状态（34 行 Zustand 风格）

**前端对应**：将 API 客户端、配置、分析工具放在 React context 之外。只有 UI 状态需要响应式。

### 3. Sticky Latches

一旦会话开始，某些决策永远不变。Beta headers 不变。Feature flags 在启动时快照。改变会无效缓存。

**前端对应**：API 版本、CDN 端点、功能开关——在应用启动时快照，不要动态重新评估。缓存稳定性 = 成本节省。

### 4. 信任边界

在用户确认信任目录之前，只有安全操作运行。之后，全部功能可用。

**前端对应**：在用户接受 cookie/权限之前，只加载必要的 JS。后续延迟加载完整应用。

### 5. 幂等 init()

`init()` 缓存其结果——多次调用是安全的。多个入口点都可以调用它，但只运行一次。

**前端对应**：用一个带缓存检查的 `getOrInitApp()` 包装你的应用初始化。
